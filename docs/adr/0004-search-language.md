# Search query language: KQL-subset, LIKE backing, structural cost-class warning

The `/requests` inspect API and the UI search box accept a whitespace-separated list of terms, AND'd together, with `-` prefix for negation. Terms are either field-qualified (`host:`, `path:`, `service:`, `body:`, `headers:`, `header.<name>:`, `method:`, `status:`, `source_ip:`, `correlation_id:`) or bare. Bare terms union across a fixed Tier-1 set of fields — host, path, service, and captured-request and event bodies and headers. Wildcards use `*` only — prefix (`foo*`) or substring (`*foo`, `*foo*`); quoted tokens (`"foo bar"`) preserve whitespace and treat `*` as literal. Field-qualified terms match only the named field. The backing store remains SQLite with plain `LIKE` predicates over the existing schema, plus one added index (`idx_events_request_path`) so the cost rule below holds uniformly across both tables.

We deliberately scoped out three things operators arriving from Kibana might expect: full boolean grammar (`AND`/`OR`/`NOT`/parens), an FTS5 backing for body/header search, and Tier-2 freeform fields (cookies, query string, content_type). Each was weighed against project shape and project cost.

**Why no `AND`/`OR`/`NOT`/parens.** httpcatch is a triage tool — narrow by `service:`, narrow by time range, filter by status, hunt a `correlation_id`. The AND-of-filters case covers the overwhelming majority of real queries. Roughly 500 lines of parser, JS mirror, and nested-chip rendering for `OR` (the only operator that genuinely needs nesting) is a poor payback ratio. The chosen grammar is a strict subset of KQL, so the upgrade path stays open and non-breaking: `-foo` becomes sugar for `NOT foo`; explicit `AND`/`OR`/parens layer on without invalidating any saved query.

**Why `LIKE`, not FTS5.** modernc.org/sqlite ships FTS5 and it would make body/header search roughly an order of magnitude faster on million-row stores. The decisive trade-off is semantics: FTS5 tokenises, so `billing-api` matches as `billing` AND `api` rather than as a literal byte sequence, and prefix wildcards apply to tokens rather than to substrings. Operators would have to internalise a second match model on top of the new query language. The CONTEXT.md framing — freeform is the slow grep, narrow with a time range first — works cleanly with `LIKE` and applies uniformly across indexed and scanned dimensions. FTS5 can be added later as an opt-in (e.g. `[sinks.sqlite] fts5 = true`) with a re-stated match model for that mode, without breaking the query language.

**The unindexed-scan warning surface.** Freeform tokens union over both indexed (host, path, service) and scanned (body, headers) fields, so a single query box produces both fast and slow work depending on the term. A leading wildcard on an indexed field silently bypasses its index. To make this visible without preventing it, we ship a structural warning: the **inspect API** sets `X-Httpcatch-Scan: leading-wildcard-indexed` on responses to any query whose shape forces an indexed dimension into substring matching, and the UI renders an amber "unindexed scan" banner under the search box while the property holds. The warning is per-query-shape, not per-result-size — operators learn the affordances by hitting the wall once.

## Consequences

- Operators arriving from Kibana will type `OR` and `AND` and get nothing back. The UI should detect a literal `OR` token between two field-qualified terms and surface a one-shot "use saved views to compare queries" hint. The discoverability tax is accepted in exchange for the parser-complexity savings.
- `body:` and `headers:` search scales linearly with row count. The unindexed-scan warning makes the affordance visible (time range or `service:` to narrow). Operators hitting the ceiling have one well-defined path before reaching for FTS5.
- The Tier-1 union is fixed in CONTEXT.md. Promoting a Tier-2 field (cookies, query string, content_type) later is a CONTEXT.md change plus one line in the union expansion — straightforward but semantically meaningful, so it should be discussed not slipped in.
- Per-header search depends on SQLite's JSON1 (`json_each`, `json_extract`); modernc.org/sqlite ships JSON1 by default. A future SQL-driver swap would have to re-verify this.
- `events.request_path` gains `idx_events_request_path` so freeform path matching is uniformly index-backed across both tables. This is a small extra write cost per event insert.
